# 统一实验协议（通用版）

## 1. 文档目的

这份文档用于定义当前仓库里 **所有后续时间序列预测实验** 的统一实验协议，作为新增 baseline、CiTransformer 变体、消融实验和调参实验的共同参考。

核心目标只有 4 个：

- 保证不同模型之间比较公平
- 避免数据泄漏
- 保证结果目录和导出格式一致
- 让后续实验可以直接复用已有脚本和分析流程

如果后面新增实验与这份协议不一致，建议不要“默认沿用旧口径”，而是必须在实验说明里明确写出偏离点。

---

## 2. 适用范围

这份协议适用于以下几类实验：

- 规则型 baseline
  - 例如：Persistence
- 可训练的神经网络 baseline
  - 例如：LSTM、vanilla iTransformer
- 你的正式模型与其变体
  - 例如：CiTransformer、去掉某个模块的 ablation 版本
- validation-only 调参实验

不在这份协议中的内容包括：

- 数据清洗本身的设计决策
- 论文写作措辞
- 图表排版细节

这些内容可以单独记录，但不应覆盖本协议定义的核心实验口径。

---

## 3. 当前统一固定项

后续实验默认共享以下固定项。

### 3.1 数据切分

统一使用同一套时间顺序 split：

- `train`
- `validation`
- `calibration`
- `test`

默认数据位置：

- `data/processed/train.csv`
- `data/processed/validation.csv`
- `data/processed/calibration.csv`
- `data/processed/test.csv`

如果这 4 个文件不在 `data/processed/` 根目录下，则自动回退到：

- `data/processed/splits/train.csv`
- `data/processed/splits/validation.csv`
- `data/processed/splits/calibration.csv`
- `data/processed/splits/test.csv`

对应实现：

- `utils/datasets.py`
- `scripts/run_lstm.py`

### 3.2 目标列与输入特征

统一预测目标：

- `target_col = Active_Pow`

默认输入特征：

- `Active_Pow`
- `Radiation_Global_Tilted`
- `Radiation_Diffuse_Tilted`
- `Weather_T`
- `Weather_R`
- `solar_elevation`
- `sin_time_of_day`
- `cos_time_of_day`
- `sin_day_of_year`
- `cos_day_of_year`
- `day_night_label`

默认情况下，后续模型应沿用这组特征，除非该实验本身就是特征消融或特征扩展实验。

### 3.3 默认时序窗口

当前统一默认配置为：

- `seq_len = 96`
- `pred_lens = [1, 12, 24, 48]`

这意味着：

- 同一批正式对比实验，默认应在这 4 个预测步长上分别导出结果
- 如果只跑其中一部分，必须在实验说明中明确写出原因

### 3.4 默认结果维度

所有正式结果默认都要汇报两套主指标：

- `all_timestamps`
- `daytime_only`

并保留逐 horizon 指标：

- `per_horizon_all`
- `per_horizon_daytime`

## 4. split 的统一语义

### 4.1 train

用途固定为：

- 拟合 `feature_scaler`
- 拟合 `target_scaler`
- 训练模型参数
- 推断默认采样间隔 `expected_delta`

### 4.2 validation

用途固定为：

- 训练过程中的验证集损失监控
- best checkpoint 选择
- early stopping
- validation-only 调参时的模型比较依据

### 4.3 calibration

当前默认协议中：

- 会被读取
- 会进入数据统计
- 但 **默认不参与训练、不参与 checkpoint 选择、不参与最终主结果导出**

因此，如果某个实验没有真的使用 calibration，应该统一写成：

- `calibration_usage = loaded_but_unused`

如果以后某类实验真的使用 calibration，也必须在说明中单独解释：

- 为什么要用
- 用在什么阶段
- 是否影响了和旧实验的可比性

### 4.4 test

用途固定为：

- 只在最终配置确定后做最终导出
- 不参与超参数选择
- 不参与 checkpoint 选择

这一点对正式论文实验非常重要：**test 只能用于最终汇报，不能回流参与调参。**

---

## 5. 无泄漏数据协议

### 5.1 scaler 规则

统一要求：

- `feature_scaler` 只在 `train` 上 `fit`
- `target_scaler` 只在 `train` 上 `fit`
- `validation / calibration / test` 只允许 `transform`

禁止做法：

- 在全数据上 fit scaler
- 先拼接 train 和 validation 再 fit scaler
- 让 test 信息参与任何标准化参数估计

### 5.2 样本构造规则

统一使用“基于连续时间片段”的滑窗协议：

1. 先按时间戳排序
2. 从 `train` 时间戳推断默认采样间隔，或由 `sampling_freq_minutes` 显式指定
3. 如果相邻时间点间隔不等于预期采样间隔，则开始新的 segment
4. 滑窗只允许在同一个连续 segment 内部构造
5. 长度小于 `seq_len + pred_len` 的 segment 直接丢弃

禁止做法：

- 直接对整张表按 `iloc` 连续滑窗
- 让样本跨越时间断点
- 对缺失段前后的数据强行拼接

对应实现：

- `utils/datasets.py`

### 5.3 指标计算尺度

统一要求：

- 训练过程中的 `loss` 可以在标准化空间计算
- 最终导出的 `validation_metrics`、`reported_metrics`、`test_metrics` 必须在 **inverse transform 之后的原始功率尺度** 上计算

因此：

- `history.train_loss`
- `history.validation_loss`

不能直接与导出结果里的 `MAE / RMSE / MSE` 混为同一个量纲。

## 6. 模型训练与选择规则

### 6.1 对可训练模型

默认训练流程固定为：

1. 在 `train` 上训练
2. 每个 epoch 在 `validation` 上计算验证损失
3. 按 `validation loss` 保存 best checkpoint
4. 用 early stopping 控制训练终止
5. 训练结束后重新加载 best checkpoint
6. 再导出 validation / test 结果

这适用于：

- LSTM
- vanilla iTransformer
- 后续多数神经网络模型

### 6.2 对无训练 baseline

对于无参数或无训练过程的 baseline，允许没有训练阶段，但仍必须遵守相同的数据与评估口径：

- 仍然只在 `train` 上 fit scaler
- 仍然在 `validation` 和 `test` 上分别导出结果
- 仍然输出统一的 `metrics.json / predictions.csv / pred_plot.png`

这适用于：

- Persistence

### 6.3 checkpoint 选择与调参排序要分开

统一建议：

- 单个 run 内部：按 `validation loss` 选 best checkpoint
- 多个超参数组合之间：可以按更贴近任务目标的 validation 指标排序

例如：

- iTransformer tuning 当前优先看 `daytime_only RMSE / MAE`

但无论如何：

- **超参数排序必须只看 validation，不看 test**

---

## 7. 统一指标协议

### 7.1 主指标

每个实验都应至少导出以下指标：

- `MAE`
- `MSE`
- `RMSE`
- `sMAPE`
- `MAPE(nonzero)`

其中：

- `MAPE(nonzero)` 只对 `abs(y_true) > 1e-6` 的样本计算
- 这是为了避免夜间零功率导致除零问题

### 7.2 两套主视角

所有正式实验默认同时导出：

- `all_timestamps`
- `daytime_only`

如果论文正文只重点讨论其中一套，也建议把另一套完整保留在附表或补充材料中。

### 7.3 白天指标的解释优先级

对于光伏功率预测：

- `all_timestamps` 适合完整汇报
- `daytime_only` 更接近真实发电预测难度

因此后续实验如果需要设定“主要优化目标”，优先考虑白天指标是合理的，但要在文档里明确写出：

- 是不是只在排序阶段偏向 daytime
- checkpoint 选择是否仍然按 validation loss

## 8. 统一输出目录结构

### 8.1 正式实验

统一推荐结构：

```text
results/<experiment_name>/
  pred_len_1/
    metrics.json
    predictions.csv
    pred_plot.png
  pred_len_12/
    metrics.json
    predictions.csv
    pred_plot.png
  pred_len_24/
    metrics.json
    predictions.csv
    pred_plot.png
  pred_len_48/
    metrics.json
    predictions.csv
    pred_plot.png
```

checkpoint 推荐结构：

```text
checkpoints/<experiment_name>/
  pred_len_1/best_model.pth
  pred_len_12/best_model.pth
  pred_len_24/best_model.pth
  pred_len_48/best_model.pth
```

### 8.2 validation-only 调参

统一推荐结构：

```text
results/tuning/<experiment_name>/<plan>/
  pred_len_<x>/<run_name>/
    metrics.json
    predictions.csv
    pred_plot.png
  summary/
    stage1/
    final_validation/
```

这类实验的 `report_split` 应为：

- `validation`

并且：

- 不加载 test
- 不导出 test_metrics
- 不让 test 参与任何排序

---

## 9. 统一结果文件格式

### 9.1 `predictions.csv`

统一要求至少包含以下列：

- `sample_id`
- `input_start_time`
- `input_end_time`
- `target_start_time`
- `target_end_time`
- `y_true_t+1 ... y_true_t+pred_len`
- `y_pred_t+1 ... y_pred_t+pred_len`
- `day_night_t+1 ... day_night_t+pred_len`

这样后续做：

- 逐 horizon 误差分析
- 白天筛选
- 可视化

都会比较方便。

### 9.2 `metrics.json`

后续新实验建议统一保留以下顶层字段。

必需字段：

- `config`
- `experiment_name`
- `tuning_stage`
- `device`
- `baseline_type`
- `baseline_definition`
- `best_epoch`
- `best_validation_loss`
- `expected_delta_minutes`
- `dataset_summary`
- `raw_split_rows`
- `target_feature_index`
- `trainable_parameter_count`
- `report_split`
- `calibration_usage`
- `validation_metrics`
- `reported_metrics`
- `test_metrics`
- `history`
- `checkpoint_path`

说明：

- 没有意义的字段可以写 `null`
- 没有训练过程时，`history` 可以写 `[]`
- 没有 checkpoint 时，`checkpoint_path` 可以写 `null`
- `reported_metrics` 表示当前导出的主结果
  - 普通正式实验里通常等于 `test_metrics`
  - validation-only 调参里通常等于 `validation_metrics`

### 9.3 指标子结构

`validation_metrics`、`reported_metrics`、`test_metrics` 建议统一包含：

- `all_timestamps`
- `daytime_only`
- `per_horizon_all`
- `per_horizon_daytime`
- `mape_note`

## 10. 允许变化项与不允许变化项

### 10.1 可以变化的项

以下内容可以作为实验变量：

- 模型结构
- 训练超参数
- `seq_len`
- 特征集合
- 是否做小规模调参
- checkpoint 选择或 run 排序时更偏向哪一类 validation 指标

### 10.2 默认不应变化的项

以下内容如果变化，会直接影响实验可比性，因此默认不应变化：

- 数据 split
- scaler 的 fit 范围
- 时间连续 segment 规则
- 最终指标必须在原始功率尺度计算
- `all_timestamps + daytime_only` 双视角导出
- `predictions.csv / metrics.json / pred_plot.png` 的结果结构

如果这些项真的要改，必须在实验说明里明确写：

1. 改了什么
2. 为什么改
3. 会影响哪些旧结果的可比性

---

## 11. 后续新增实验的命名建议

### 11.1 实验目录名

建议：

- `results/lstm/`
- `results/persistence/`
- `results/itransformer/`
- `results/citransformer/`
- `results/citransformer_abl_no_pcmci/`

命名原则：

- 一眼能看出模型或变体是什么
- 不要把关键差异藏在模糊名字里

### 11.2 调参 run 名

建议把以下关键信息编码进 `run_name`：

- stage
- pred_len
- learning_rate
- batch_size
- 关键模型容量参数
- dropout
- weight_decay

这样后面看目录时不用再打开 `metrics.json` 才知道大概配置。

---

## 12. 后续新增实验的最小 checklist

每次新增实验前，建议逐条检查：

1. 是否使用了同一套 `train / validation / calibration / test` split
2. 是否只在 `train` 上 fit scaler
3. 是否禁止跨时间断点滑窗
4. 是否使用了统一的 `seq_len / pred_len` 集合
5. 是否至少导出了 `all_timestamps` 和 `daytime_only`
6. 是否把最终指标放在原始功率尺度上计算
7. 是否把结果写到统一目录结构
8. 是否导出了统一格式的 `metrics.json`
9. 是否避免让 test 参与调参
10. 如果协议有偏离，是否在实验说明里明确记录

## 13. 当前仓库里的参考实现

如果后续要新增模型，优先参考这些现有实现：

- 数据协议与评估共用逻辑：
  - `utils/datasets.py`
  - `scripts/run_lstm.py`
- 规则型 baseline 参考：
  - `scripts/run_persistence.py`
- 可训练 baseline 参考：
  - `scripts/run_lstm.py`
  - `scripts/run_iTransformer.py`
- iTransformer validation-only 调参参考：
  - `scripts/tune_itransformer.sh`
  - `scripts/summarize_itransformer_tuning.py`

最推荐的做法不是“另起一套新协议”，而是：

- 尽量复用现有数据准备、评估和导出函数
- 只替换模型主体
- 只在确实必要时增加新的实验变量

---

## 14. 一句话版本

后续所有实验，默认都应遵守这条总原则：

**同一套 split、同一套 train-only scaler、同一套连续 segment 滑窗、同一套 `seq_len/pred_len`、同一套 daytime/all 指标、同一套结果导出结构。**

如果某个实验偏离这条原则，必须显式记录，而不能默认混在同一张对比表里。
